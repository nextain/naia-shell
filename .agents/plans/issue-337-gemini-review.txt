1. **Master password lifecycle**: A static 32-byte master password stored in the OS keyring without rotation is completely acceptable. Adding rotation complexity introduces risk without substantial security gain, as compromising the OS keyring bypasses local file encryption regardless. Correctness and simplicity are more valuable here than over-engineered rotation mechanisms.

2. **`BackupCapable` reuse**: The ~50ms cost of PBKDF2 is negligible since it only occurs on boot, login, and rare refresh events. Reusing the well-tested `BackupCapable` implementation guarantees cryptographic correctness and structural consistency without introducing new bugs. Do not build a thinner wrapper unless empirical profiling proves this ~50ms causes unacceptable UI blocking.

3. **State token binding**: Accepting that the user must retry login if the agent crashes is the right, pragmatic choice. Storing transient OAuth state on disk adds unnecessary I/O complexity and risks state corruption for an edge case that should be exceptionally rare. Keep it memory-bound to maintain a simple and reliable state machine.

4. **`auth_query` race**: Show a explicit "checking..." state (or loading skeleton) during the ~100ms decryption window. Assuming a logged-in state and rolling back causes jarring UI jitter and risks triggering authenticated UI side-effects that will ultimately fail. Correctness and predictable UX must supersede the desire for instant, speculative rendering.

5. **gatewayToken scope decision**: Strictly defer the `gatewayToken` migration to a subsequent, separate issue. Bundling it expands the scope unnecessarily and risks breaking the WebSocket flow while you are still stabilizing the HTTP auth flow. Isolate the variables to ensure this migration is rigorously and safely verified.

6. **Migration backout**: If the agent fails to acknowledge the migration, hard-fail and require a new login. Silently falling back to the legacy shell store creates a split-brain scenario where the agent and shell disagree on the source of truth. Failing visibly guarantees consistency and prevents subtle, long-tail data synchronization bugs.

## Additional concerns

- **Missing PKCE**: The design mentions OAuth `state` for CSRF but omits PKCE (Proof Key for Code Exchange). For a native desktop application capturing a deep-link callback, standard OAuth 2.0 requires PKCE to prevent malicious local apps from intercepting the `naia://` redirect and stealing the authorization code.
- **Refresh Token Race Conditions**: While a single-flight mutex is planned for the 401 retry loop, ensure that incoming chat requests properly await the *new* token from the in-flight refresh rather than immediately failing or triggering duplicate refresh requests.
- **File Read/Write Concurrency**: Atomic writes (via `.tmp` and rename) prevent file corruption, but concurrent reads during a token refresh event must be handled gracefully so they do not read a partially swapped state or miss the new key.
- **OS Keyring Availability**: On Linux, `libsecret` may not be available on headless or minimal desktop environments, causing keyring operations to fail. Ensure there is explicit error handling or a secure fallback if the OS keyring is locked or inaccessible.
