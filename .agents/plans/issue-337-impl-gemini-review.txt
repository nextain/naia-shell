I have completed the architectural and security code review for issue #337 (ADK-centric session persistence). My audit covered implementation commits across `naia-agent` (e675a47..60c981c) and `naia-os` (d4790c57..b779ef04).

## Summary
The implementation successfully shifts the Source of Truth for authentication from the shell's `secure-keys.dat` to an agent-managed encrypted store in the ADK path. Security mandates regarding `naiaKey` isolation are strictly followed: the shell never touches the key at runtime, and the agent uses a robust AES-256-GCM envelope with PBKDF2. No critical security regressions (leaks) were found.

## Findings

### Security & Architecture

1.  **[MEDIUM] `naia-agent/packages/runtime/src/utils/keyring.ts:165`**: macOS `security` CLI usage for `add-generic-password` uses `-w [password]`, which exposes the master password in the process list (`ps` output).
    *   **Recommendation**: Use a temporary file or stdin if the local `security` version supports it, or accept this as a known transient risk for the master password (not the `naiaKey`).

2.  **[MEDIUM] `naia-os/shell/src/lib/config.ts:465`**: Known functional regression in WebSocket voice features (`gemini-live`, `naia-talk`). Fresh logins after Phase 6c do not persist the `naiaKey` to the shell's secure store, causing these features to fail for new users.
    *   **Recommendation**: Prioritize issue #338 (moving WebSocket auth into the agent) to restore these features for new sessions.

3.  **[MEDIUM] `naia-agent/packages/runtime/src/auth/ipc-handlers.ts:178`**: `handleLabProxyRequest` hardcodes `JSON.stringify` for request bodies. This breaks `multipart/form-data` uploads, which is why `api-stt.ts` (Nextain transcription) remains unmigrated.
    *   **Recommendation**: Extend the proxy to handle `Buffer` or `ReadableStream` for `POST` bodies to allow STT migration.

4.  **[LOW] `naia-agent/packages/runtime/src/utils/keyring.ts:124`**: Windows PowerShell backend allocates memory for credentials via `Marshal::AllocHGlobal` but lacks explicit `try-finally` blocks for cleanup on script errors.
    *   **Recommendation**: Use `finally` in PowerShell to ensure `FreeHGlobal` is called even if the credential write fails.

5.  **[NIT] `naia-agent/packages/runtime/src/auth/ipc-handlers.ts:154`**: Default `userId` for legacy migration is `""`.
    *   **Recommendation**: Ensure downstream logic (e.g., usage tracking) handles empty strings or defaults to a generic `naia_user` format.

### Correctness & Reliability

6.  **[LOW] `naia-os/shell/src/lib/legacy-migration.ts:135`**: Migration policy for "Agent already logged in" silently purges the `naiaKey` slot. While correct for #337, if the user had different accounts in the shell vs. agent, the shell's account is lost without warning.
    *   **Recommendation**: Given the current 1:1 user-to-device mapping, this is acceptable, but consider a "conflict" UI if multi-account support is added.

7.  **[LOW] `naia-agent/packages/runtime/src/utils/oauth-flow.ts:45`**: `naia://` scheme replacement with `https://` for URL parsing assumes the deep-link always follows the `naia://auth?...` pattern.
    *   **Recommendation**: Safe for now, but a more robust custom scheme parser would prevent edge cases with complex paths.

---

### Verified Correct (High Confidence)
*   **Zero-Leak IPC**: Verified that `auth_received`, `auth_query`, and `lab_proxy_response` explicitly strip `naiaKey`, `refreshToken`, and `issuer` before sending JSON to the shell.
*   **Refresh Single-Flight**: The `inflight` Map in `refresh.ts` correctly joins concurrent 401 retries to a single promise, preventing token rotation races.
*   **Atomic Migration**: Shell purges the legacy slot only after the agent issues a success ACK for `auth_legacy_migrate`, preventing data loss on disk/timeout errors.
