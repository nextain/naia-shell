naia 당신의 AI를, 당신의 기계에서. 코드로 검증 가능하게.

겉모습은 아바타. 본질은 함께 일하는 AI 동료.

제품 자랑으로 시작하지 않는다. naia가 무엇인지 한 줄. "버튜버처럼 보이지만 개발 동료"라는 반전은 12장에서 푼다. 여기선 톤만 — 소유·로컬·검증가능.

그런데 — 오픈소스가 AI에 지쳤습니다

  • curl, 7년 운영한 버그바운티를 2026년 1월 닫았습니다. 이유: AI가 만든 가짜 보안 제보 홍수.
  • 2025년 제보의 20%가 AI slop, 진짜 취약점은 5%.
  • "the mind-numbing AI slop, humans doing worse than ever." — Daniel Stenberg
  • 2025년 올해의 단어 = "slop"(Merriam-Webster) / "AI slop"(Macquarie)
  • Gentoo·NetBSD·QEMU는 AI 코드 기여 금지. Linux·LLVM·Fedora는 Assisted-by: + 사람 검토 의무화.
청중이 다 공감하는 현실부터. 숫자는 curl 본인 블로그 기준. "여러분도 AI slop PR 받아보셨죠?"

그들이 막는 건 AI가 아닙니다 — 검증 없는 AI입니다

"검토 안 된 LLM 출력을 메인테이너에게 보내는 것은 그들의 작업(설계·리뷰)을 추출하는 것이다." — LLVM AI Tool Policy RFC

문제의 핵심은 검토 부담의 전가. AI 자체가 아니라, 거를 게이트가 없다는 것.

이 한 장이 발표의 축. 적개심의 대상을 정확히 짚으면 우리 포지션이 선다.

빈자리: "AI가 만들되 게이트로 품질을 보증하는" 모델

  • 인기 GitHub 1,000개 레포 조사(arXiv 2605.16706, 2026): 공식 AI 정책은 118개뿐, 전부 "환영~금지"의 방어 스펙트럼.
  • 저자 결론: "AI-native로 규정·운영되는 프로젝트는 없다."
  • 가장 근접한 Aider도 AI 작성 비중을 공개할 뿐, 계약·게이트로 slop을 능동 차단하진 않습니다.

세상은 "AI 기여를 어떻게 막을까"를 묻습니다. 우리는 "AI로 만들되 slop을 어떻게 막을까"를 풉니다.

"세계 최초"라고 말하지 않는다. "선례가 드문 반대편"이라고만. 정직이 신뢰를 만든다.

우리 답 — 흔들림 없는 구성

AI로 빠르게 만든다. 단, 깨지지 않게.

slop을 사람 의지가 아니라 구조(게이트)로 막는다.

다음 장들이 이 한 문장의 실제 구현. 추상 구호로 끝내지 않고 바로 데모로 간다.

"깨지지 않음"의 정체 — slop 5가지

  • ① 사전 합의 없는 파일/폴더 생성 (루트뿐 아니라 전 계층)
  • ② 컨텍스트 드리프트 — 어느새 전혀 다른 방식으로 구현
  • ③ 추적 안 되는 고아 파일
  • ④ 고아/합의 없는 함수
  • ⑤ 지켜지지 않는 코딩 컨벤션
추상적 "AI slop 방지"가 아니라 구체적으로 이 5가지. 청중이 "맞아 저거"라고 느끼게.

헥사고날 — 경계가 곧 게이트

adapters  →  ports  →  domain
(바깥/구현)   (계약)     (안/순수)
  • 의존은 한 방향만. domain은 아무것도 모른다.
  • 위 5가지가 하나의 메커니즘으로 잡힘: 위치=① · 의존방향=② · 미연결 노드=③④ · 컨벤션=⑤
  • 제공자(TTS·LLM, 나중엔 naia 런타임)는 어댑터로 교체 — 코어는 안 바뀝니다.
"다 추적"은 파일마다 등록(무겁다)이 아니라 규칙이 추적한다. 기여자는 규칙대로 짜면 통과.

추적 체인 — 무엇이 무엇과 묶여 있나

이슈(의도) → 문서(설계) → 코드(포트 interface) → 테스트(계약)

2층 계약 SoT (각 계약은 한 곳에만 → 갈라질 수 없음):

  • 용어·의도·불변식 → 문서(단어사전)
  • 시그니처·타입 → 코드의 interface (코드가 곧 계약)
  • 코드 주석엔 계약을 다시 쓰지 않고 문서를 가리키기만.

→ "이 코드 고치면 뭐가 깨지지?"의 답이 항상 추적 가능.

회귀(수정→다른 데 깨짐)가 빈번한 이유 = 추적이 끊겨서. 이게 그 해법.

데모 ① — 하나의 계약, 여러 어댑터 (그리고 이식)

목업과 실제(Edge TTS — 기존 코드에서 이식)가 같은 계약을 통과:

✓ TtsPort contract: MockTtsAdapter (2 tests)
✓ TtsPort contract: EdgeTtsAdapter (2 tests)   ← 기존 코드 선별 이식
  Test Files  2 passed (2)
       Tests  4 passed (4)
라이브로 pnpm test. 계약 러너에 어댑터 한 줄 추가 = 기존 코드 이식. 같은 계약이 mock·real을 동시에 지킨다. "통째 재작성"이 아니라 "계약 아래로 한 조각씩".

데모 ② — 슬롭을 넣으면 게이트가 빨개진다

일부러 위반 주입: domain이 adapter를 import하게:

1) 정상      ✔ no dependency violations (5 modules)
2) 위반 주입  ✗ domain-stays-pure: _probe.ts → adapter   ← 게이트가 잡음
3) 되돌림     ✔ no dependency violations
이게 절정. 슬롭을 심으면 자동으로 RED, 빼면 GREEN. 사람이 안 봐도 막힌다.

왜 다른가 — 자기 선언이 아니라 반증 가능

"우리 코드 깨끗합니다"(자기 선언) ✗

→ "슬롭을 심어봤더니 게이트가 RED로 잡더라"(반증 가능) ✓

AI가 통과시키려 해도 게이트는 반증을 겨냥합니다.

우리 철학 한 줄 — "가짜를 스스로 RED로 만들 수 있는가." 이게 검증의 기준.

그래서, naia란

  • 개발자가 소유하는 자기만의 AI. 내 기계, 내 데이터, 코드로 검증 가능.
  • 겉은 아바타지만 본질은 함께 일하는 동료 — 이슈·터미널·코드를 같이 다룬다.
  • 클라우드 종속이 아니라 로컬·프라이버시 우선.
  • 그리고 이 naia를, 방금 보여준 그 구성으로 만듭니다. (dogfooding)
1장의 반전을 여기서 회수. naia는 컴패니언 단일 제품이 아니라 "소유"가 쐐기. 과장 금지.

함께 만들자 — 기여는 이렇게

  • 흐름: 이슈 → 문서 → 코드 → 테스트 (게이트가 빠진 칸을 알려줌)
  • 두 갈래 트랙: quick-fix(가벼운 수정) / feature(시나리오부터 풀 추적)
  • 당신의 AI(codex·claude…)를 데려와도 됩니다 — 게이트가 slop을 거르니까.
  • AI 사용은 정직하게 표기(Assisted-by:), 과정은 샘플링 감사로.
진입장벽 낮추기. "AI 써도 된다, 단 게이트를 통과해야 한다"가 차별점.

로드맵 — 정직하게

지금 (있음)나중 (방향)
헥사고날 코어 + 의존성 게이트코드그래프(코드에서 추출하는 살아있는 지도)
TTS 포트 + 계약 테스트어댑터로 naia 자체 런타임 교체
추적 체인 + 2층 계약 SoT의미 드리프트 3자 매핑
안 만든 걸 만든 척하지 않는다. "방향"과 "있음"을 명확히 가른다. 이 정직함이 곧 데모의 신뢰.
