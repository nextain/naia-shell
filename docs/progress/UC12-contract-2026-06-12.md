## B.2 ports/ (기존 index.ts **확장**, 재발명 금지 — codex R3)
**driving (신규) — control-plane config 인바운드 (R2-4: 정의된 인터페이스가 소유. R12-2: 온보딩·설정 두 driving 연산을 명시 — "단일 인터페이스" 아님). `OnboardingFlowPort`(온보딩) + `SettingsPort`(완료후 편집) 둘 다 driving 표면**:
- `current():OnboardingState`
- `submit(step,input:StepInput):Promise<OnboardingState>` — 단계 제출 → advance. ⚠️ **`StepInput` = 단계별 타입드 union의 *resolved 값***(R15-2: 순수 `advance`는 I/O 없음 — asset 목록 fetch/기본선택은 controller 가 `assets()` 로 미리 해 선택된 값을 input 으로; advance 는 vrmModel/background 등 *확정 값*만 받음). 빈 목록 시 미설정(controller 판정).
- `assets(kind:'vrm-files'\|'background'):Promise<AssetRef[]>` — character/background 단계용(R14: UI 가 driven AssetInventoryPort 직접 호출 금지 — 이 driving 포트가 노출, controller 가 AssetInventoryPort+AdkPathPort 조합).
- `startNaiaAuth():Promise<void>` — provider 단계 naia OAuth launch(R14: UI→driving 만; controller 가 driven OAuthPort.launch 호출). callback 은 onNaiaAuthCallback(아래).
- ⚠️ **OAuth 경계**(R15-1): `OAuthPort.launch()` = 플로우 개시만. `onNaiaAuthCallback(payload:{naiaKey})` = **payload 가 resolved `naiaKey` 운반**(baseline: callback event.payload.naiaKey 직접 전달 — code 교환 없음). 키 영속 소유 = controller(writeAgentKey, NAIA_ANYLLM_API_KEY). code 교환형이면 OAuthPort.exchange(code)→naiaKey 가 controller 앞단(이 슬라이스는 baseline=resolved naiaKey).
- `onNaiaAuthCallback(payload):Promise<OnboardingState>` — **이 포트 소유 inbound 연산**(driven 아님). naia OAuth callback 수신 시: ⚠️ **idempotent**(R2-5) — `naiaLoginDone===true` 면 추가 write/authUpdate 없이 현 state 반환(중복 callback no-op). 아니면 naiaKey(top-level) 영속 + `GatewaySyncPort.authUpdate` **즉시 1회**(완료와 무관 — R6) + naiaLoginDone=true + memory provider 자동 "naia"(값) → 갱신 state 반환.
- `complete():Promise<void>` (아래 영속 순서).
- **`SettingsPort.update(patch):Promise<void>`** (R5-1: baseline S02/S03 = 온보딩 후 설정 편집). ⚠️ **merge base = `BootStatePort.loadLocalConfigWithSecrets() ?? baseConfig()`**(기존 — secret 포함 read; R8-1: null 이면 기존 `baseConfig()`={agent:{},secret:{},ui:{}} 폴백. R6-1: 키 외 변경 시 기존 apiKey/naiaKey 보존). ⚠️ **per-category 깊은 병합**(R7-1: shallow `{...base,...patch}` 는 agent/ui/secret 통째 교체로 미패치 키 유실) — `merged = { agent:{...base.agent,...patch.agent}, ui:{...base.ui,...patch.ui}, secret:{...base.secret,...patch.secret}, naiaKey: patch.naiaKey ?? base.naiaKey, workspaceRoot/onboardingComplete 보존 }` → 동일 영속(stripSecret→replaceLocalConfig + ConfigPort.write(adkPath,forAgent) + 변경된 secret→CredentialStore.writeAgentKey overwrite + GatewaySync.sync). **S03 provider 전환**(R7-2): provider→envKey 매핑 = **도메인 소유 `resolveAgentEnvKey`(domain/config.ts, R10)**. 전환 시 **구 provider 의 envKey 해석 → writeAgentKey(oldEnvKey,"") 로 비움** + 신 provider 키 set. ⚠️ **provider 변경 시 `secret.apiKey` 미보존**(R12-1: 직전 provider 키를 새 provider envKey/cfg/GatewaySync 에 흘리지 않음 — provider 가 바뀌면 secret.apiKey 리셋 후 신 키만; 키 없는 provider 면 apiKey 없음). naiaKey(naia 분기)는 별도 유지. (진짜 delete op = CredentialStorePort deferred 후속.) onboardingComplete 게이트 무관.

**영속 = 기존 포트 그대로 (신규 write 금지)**:
- 로컬 config: `BootStatePort.replaceLocalConfig(stripSecret(cfg))` — secret 은 CredentialStore 전담, 로컬엔 secret 미포함. ⚠️ **`stripSecret(cfg):NaiaConfig` = `secret:{}` + top-level `naiaKey` 제거**(R4-1: naiaKey 도 secret이므로 CredentialStore 전담 — 로컬 영속 금지; 기존 baseline secure-store 정합). secret category + naiaKey 둘 다 비움, 나머지 보존. 타입 NaiaConfig 유지. domain 순수 projection.
- agent-config 파일: `ConfigPort.write(adkPath, forAgent(cfg))` (기존 시그니처 — secret+ui strip).
- secret: `CredentialStorePort.writeAgentKey(envKey, value)` (기존). ⚠️ **envKey = `resolveAgentEnvKey(provider, keyField)` — 이식 시 이 매핑을 `domain/config.ts`(순수, import 0)에 정의**(R10-1: 현 baseline 은 concrete adk-store.ts 소유라 app→concrete 의존 위반; 도메인 소유 순수함수로 옮겨 app/ports 만 의존). 매핑: `apiKey`→anthropic→ANTHROPIC_API_KEY·openai→OPENAI_API_KEY·glm→GLM_API_KEY, `naiaKey`→NAIA_ANYLLM_API_KEY. ⚠️ **키 없는 provider = 의도적 null**(R11-1, 기존 동작): gemini(naia-cloud 경유)·ollama/vllm(open)·claude-code-cli → resolveAgentEnvKey=null → **writeAgentKey 미호출**(영속할 직접 키 없음). onboarding complete·SettingsPort·OAuth callback 모두 이 도메인 매핑 사용(null=skip).

**driven (신규)**:

## §D 신규 계약 델타 (2026-06-14) — `completeWith()` 셸-graft seam

**배경(GOAL⑥ "구현 불가능 시 신규 계약")**: 기존 `OnboardingController.complete()` 는 컨트롤러 내부 `state.draft`(submit() 으로 단계별 구동)에서 persist 한다. 그러나 실 셸 `OnboardingWizard.tsx`(770줄, 자체 snapshot 상태)는 **외부에서 완성 config 를 빌드**한 뒤 한 번에 저장하는 패턴 — 컨트롤러를 단계별로 구동하지 않는다. 따라서 `complete()` 로는 빈 draft 가 persist 되어 그래프트 불가. **현 계약으로 구현 불가 → 신규 계약**.

**계약**: `completeWith(cfg: NaiaConfig): Promise<void>`
- 외부에서 빌드한 `NaiaConfig`({agent,secret,ui,naiaKey?,workspaceRoot?,onboardingComplete}) 를 받아 **기존 `persist()` 경로 그대로**(replaceLocalConfig(stripSecret)+config.write(forAgent)+키체인 envKey) 영속 + `bootState.markOnboardingComplete()`.
- = `complete()` 와 동일 부작용(영속 규칙·키체인·markComplete)이되 source 가 내부 draft 가 아닌 인자.
- **불변식 유지**: secret 은 로컬 미포함(키체인 전담, R4/R6/R8 = UC12 리뷰 stale-credential fix). provider 전환 시 구 키 clear 는 `update(providerChanged)` 경유(completeWith 는 신규 온보딩 완료 = 전체 교체라 해당 없음).
- **graft**: 셸 `OnboardingWizard.handleComplete` 가 `isNewCore()` 시 wizard snapshot → NaiaConfig 매핑 후 `makeShellOnboarding(...).completeWith(cfg)` 호출. 미설정(기본)=기존 saveConfig/writeNaiaConfig 경로 보존(비파괴). UC1 chat-service graft seam 과 동일 패턴.
- **검증**: Playwright e2e(`__NAIA_NEW_CORE__=true`)로 wizard 완료 → write_naia_config/write_agent_key 발신 확인. + 단위(controller.completeWith persist 부작용).
