# §A. Old-Baseline (측정 = UC12-baseline-2026-06-12.md)
- 8단계: `welcome→agentName→userName→speechStyle→character→background→provider→complete` (단계↔필드 매핑 = baseline 표).
- 산출 = `AppConfig` 표면(신원/표현·provider/model·키·게이트) + `onboardingComplete=true` + `syncToGateway`.
- 자산(VRM/배경) = `{adkPath}/naia-settings/{vrm-files,background}` (`listNaiaAssets`). provider 단계 naia OAuth 분기.
- **판정**: 8단계 플로우·산출 config = **이식**(old-auth 동작). 포트 추상화(직결 UI→포트) = **보충**. memory(memoryEmbedding/Llm) provider 선택 = **다른 세션 경계** — 본 계약은 *필드 보존*만(값 produce), naia-memory 연결 로직 미포함.

# §B. 포트 계약
## B.1 domain/ (순수, import 0)
| 값객체 | 규칙 |
|--------|------|
| `OnboardingStep` | 폐쇄 union: `welcome\|agentName\|userName\|speechStyle\|character\|background\|provider\|complete` (순서 = `STEPS` 배열, 권위). |
| `OnboardingState` | `{step, draft:Partial<NaiaConfig 표면>, naiaLoginDone}` 단계 진행 상태기계. `advance(state, input)→state'`(순수): 각 단계가 draft 의 해당 필드 set(매핑=baseline 표), 마지막 complete 에서 draft→완성 config. **단계 건너뜀 금지**(순서 불변식). |
| `NaiaConfig`(기존 확장) | 현재 최소(onboardingComplete 등) → **AppConfig 표면 필드 추가**(agentName·userName·speechStyle·vrmModel·background·provider·model·naiaGatewayUrl·locale·theme + memory* 필드는 *보유만*). secret(apiKey/naiaKey)=도메인 보유, 송신은 creds 별채널(F0 stripForAgent 정합). |
| `completeOnboarding(draft)→NaiaConfig` | 순수: draft + `onboardingComplete=true`. |

## B.2 ports/ (driven+driving) — 기존 index.ts 확장
- `OnboardingFlowPort`(driving — shell UI 가 호출): `current():OnboardingState` · `submit(step, input):OnboardingState` · `complete():Promise<void>`(=ConfigPort.write + GatewaySyncPort.sync). 8단계 상태기계 노출.
- `ConfigPort`(기존, 확장): `read(adkPath):NaiaConfig|null` + **`write(cfg:NaiaConfig):Promise<void>`**(localStorage "naia-config" + naia-settings/config.json NAIA_MAIN_* 변환=buildNaiaConfigEnv 정합). 기존 read 보존.
- `AssetInventoryPort`(driven): `list(kind:'vrm-files'\|'background'):Promise<AssetRef[]>` (= `{adkPath}/naia-settings/{kind}`). character/background 단계 입력원.
- `AuthCallbackPort`(driven): provider 단계 naia OAuth callback → `naiaKey` + `naiaLoginDone`. (기존 CredentialStorePort 와 연계, 신규 OAuth 수신만.)
- `GatewaySyncPort`(driven): `sync(cfg)` / `authUpdate(naiaKey)` (= 기존 syncToGateway/sendAuthUpdate). complete 시 1회.
- `CredentialStorePort`(기존): secret 영속. 재사용.

## B.3 app/ (조립 — 구상 직접의존 금지, 포트만)
- `OnboardingController` = OnboardingFlowPort 구현. domain `advance` 로 상태기계 구동, complete 시 ConfigPort.write + GatewaySyncPort.sync + AuthCallbackPort(provider 단계). boot 결정(decideBoot, 기존)과 연계: onboardingComplete → Main.

# §C. acceptance (e2e-tauri 로 검증 — 유저 테스트 가능)
1. 8단계 순서 = baseline `STEPS` verbatim (건너뜀 0).
2. 각 단계 산출 필드 = baseline 매핑 표와 일치(GREEN이어도 다르면 FAIL).
3. complete → ConfigPort.write(full config) + onboardingComplete=true + GatewaySync 1회.
4. provider 단계 naia OAuth 분기 보존(naiaKey 저장 → memory provider 자동 naia *값만*).
5. 실 Tauri 앱(e2e-tauri, 02-configure/configureSettings 재사용)서 온보딩→설정 플로우가 유저 동작으로 manifest.
6. 새 src/main 파일 = module-manifest 앵커 선행(file-anchor-guard 강제).
