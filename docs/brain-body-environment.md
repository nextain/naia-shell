# 뇌·몸·환경 레이어 표준 (Brain / Body / Environment)

> **목적**: naia 를 세 레이어 — **뇌(brain) · 몸(body) · 환경(environment)** — 로 자른다.
> 이 표준은 "무엇이 어디에 살아야 하는가"의 SoT 다. 위반(예: 환경 런타임이 뇌 안에 사는 것)은
> split repo·안드로이드/로봇 포팅에서 즉시 깨진다. 사상은 게이트로 강제한다(§6).
>
> 핵심 한 줄: **뇌는 의도(intent)만 emit, 몸은 감각·표현 기관, 환경은 뇌에서 분리된 독립 서비스(사이드카).**
>
> ⚠️ 용어 SoT = `docs/glossary.md`. 본 표준의 **몸 = 육체(body)**, **환경(environment)** 정의는 glossary 와 동일하다
> (glossary: "환경 = body 밖 세계 — 공간·배경·**BGM**·앱 surface·시스템, shell 이 렌더/호스트하나 body 아님").
> 이 문서는 그 모델의 **규범(무엇이 어디 살고, 환경은 어떻게 사이드카로 실현)** 을 더한다.

---

## 1. 세 레이어

| 레이어 | 정의 | repo/런타임 | 예 |
|---|---|---|---|
| **뇌 (Brain)** | **인지** — 지각·기억·추론·의도. "무엇을 할지" 결정. substrate-agnostic. | `new-naia-agent` (Node) | provider 라우팅, 도구 선택, 기억, 추론 |
| **몸 (Body)** | **에이전트 자신의 감각·표현 기관** — 보고/듣고/말하고/표현하는 채널 = 에이전트의 현존(presence) | `new-naia-os` 셸 네이티브 (Tauri/webview/오디오) | 아바타, 음성 출력(TTS/express), 음성 입력(mic/STT), 시선·표정 |
| **환경 (Environment)** | **에이전트가 그 안에서/위에서 작업하는 세계** — 도구·자원·분위기. 에이전트와 독립으로 존재. | **셸 소유 독립 서비스(사이드카)** | 브라우저, 터미널, workspace/파일, **BGM/공간 분위기**, 외부 도구 |

> **도구는 몸이 아니라 환경이다.** 브라우저·터미널·workspace 는 에이전트의 *기관*이 아니라
> 에이전트가 *조작하는 세계의 일부*다. 따라서 환경 레이어에 속하고, 환경 규칙(독립 서비스)을 따른다.

### 분류 테스트 (한 기능이 어느 레이어인가)

1. **"결정하는가?"** (인지·의도) → **뇌**. (예: "어떤 모델로? 무슨 도구를? 뭘 기억?")
2. **"에이전트 자신의 감각/표현 기관인가?"** (귀=mic, 입·얼굴=voice/avatar) → **몸**.
3. **"에이전트가 그 안에서 작업하는 세계의 일부인가?"** (브라우저·터미널·파일·음악) → **환경**.

> 애매하면 디폴트는 **환경**(가장 분리도가 높고 안전). 몸은 "에이전트의 현존 기관"으로 *좁게* 본다.

## 2. 환경 = 독립 서비스(사이드카) — 불변식

환경은 **뇌에서 분리된, substrate(셸/OS)가 소유하는 독립 서비스**다. 다음을 반드시 만족한다:

- **E1. 뇌 독립 (brain-optional)** — 환경은 뇌(agent)가 죽었거나 재시작 중이거나 아예 없어도 동작한다.
  뇌는 환경에 **intent 만** 보낸다(그것도 옵션). 환경이 뇌에 종속되면 위반.
- **E2. substrate 소유** — 환경의 **코드·deps·프로세스 수명**은 셸(데스크톱) / OS(로봇·안드로이드)가 소유한다.
  뇌(agent) 트리에 환경 런타임(서버·바이너리·deps)을 두지 않는다. (그래야 뇌가 substrate-agnostic 유지.)
- **E3. intent 계약으로만 접점** — 뇌→환경은 의도(navigate / run / play mood)만. 뇌는 "어떻게"를 모른다.

### 실현 형태 (둘 다 "환경 서비스"의 사이드카 패턴)

| 형태 | 언제 | 예 |
|---|---|---|
| **A. Rust/Tauri in-process** | 셸 Rust 가 OS 능력을 직접 제공 가능 | 터미널(PTY), 파일 관측·조작 |
| **B. 별도 프로세스 사이드카** | 셸 Rust/webview 가 못 하는 **외부 런타임** 필요 (Node lib·브라우저 엔진·외부 바이너리) | youtube 추출(youtubei.js), 브라우저 자동화(CDP 드라이버) |

> A/B 모두 **셸 소유 + 뇌 독립**(E1~E3). "사이드카로 가느냐"의 기준 = **외부 런타임이 필요한가**(→ B), 아니면 Rust 로 되는가(→ A).
> Rust 로 되는 걸 굳이 별도 프로세스로 빼지 않는다(IPC 오버헤드, 오버엔지니어링).

### 환경 사이드카 판정 (B 를 택하는 3조건 — 전부 참)

1. **환경이다** (§1 분류 테스트로 환경).
2. **셸 Rust/webview 네이티브로 불가** → Node lib / 브라우저 엔진 / 외부 바이너리 필요.
3. **뇌 재시작/부재에도 생존해야** (E1).

→ 충족 시: 셸이 spawn·소유하는 **별도 프로세스 사이드카**. 코드·deps 는 셸 쪽(`new-naia-os`). 뇌는 intent 만.

## 3. 레이어별 소유·배치 규칙

- **뇌(agent)** = 순수 인지만. **환경 런타임(서버/프로세스/외부 deps) 호스팅 금지.**
  인지 스킬(weather/github/memo/mcp/time = 네트워크·로직, substrate-agnostic)은 agent 에 둔다.
  **환경 스킬(bgm/browser/terminal)은 agent 가 실행하지 않는다** — intent 만 emit, 실행은 환경 서비스가.
- **몸(셸 네이티브)** = 아바타·음성 I/O·표정. Tauri webview/오디오 직속. 사이드카 아님(에이전트의 기관).
- **환경(셸 소유 서비스)** = 도구·분위기. A(Rust in-process) 또는 B(별도 프로세스 사이드카). 뇌 독립(§2).

| 기능 | 레이어 | 결정(intent) | 실행(런타임) | 현재 상태 | 목표 |
|---|---|---|---|---|---|

| **브라우저** | 환경(도구) | agent 스킬 (navigate/click) | 패널=셸 웹뷰(몸 아님, 환경 표면) + 자동화=CDP | ⚠️ 패널=셸(좋음) **but** 자동화를 agent 가 external CLI 로 직접 호출 | **셸 소유 브라우저 드라이버 사이드카(B)**, agent 는 intent 만 |

| **파일 관측·조작** | 환경 | agent | Rust(in-process) | ✅ Rust 네이티브(A) | 유지 |
| **아바타 / 음성 출력 / mic** | **몸** | agent (표현 intent) | 셸 네이티브 I/O | 셸/Tauri | 유지(몸 = 기관) |
| **인지 스킬** (weather/github/memo/mcp/time) | **뇌** | — | agent-local | ✅ agent | 유지(substrate-agnostic) |

## 5. 왜 이게 중요한가 (substrate-agnostic)

뇌는 데스크톱·안드로이드·로봇으로 **몸과 환경만 갈아끼우고 그대로** 가야 한다.
환경 런타임(youtube Node 서버, Windows Chrome 실행)을 뇌에 넣으면 뇌가 특정 몸(Windows 데스크톱)에 오염되어 포팅이 깨진다.
로봇엔 브라우저 패널이 없고 "환경"이 통째로 다르다 — **환경 사이드카만 교체, 뇌 불변**이어야 한다.
"뇌 죽으면 음악도 끊긴다"는 단순 가용성 문제가 아니라 **레이어 경계 위반의 증상**이다.

- 현재는 본 표준 = 수동 준수 + 리뷰 체크. 게이트 인코딩은 follow-up(§ ARCHITECTURE §5 게이트 표에 편입).
