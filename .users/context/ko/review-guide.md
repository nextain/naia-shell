모든 리뷰 게이트에서 적용하는 통합 가이드입니다. 단계별 자가 리뷰, PR 휴먼 리뷰, 적대적 크로스 리뷰 모두 이 가이드를 기준으로 합니다. **머지 전 6개 차원 모두 통과해야 합니다.**

---

## 적용 시점

| 게이트 | 트리거 | 최소 적용 차원 |
|--------|--------|--------------|
| 단계별 리뷰 | 각 빌드 단계 완료 후 | 아키텍처, 컨벤션, 요구사항 |
| 커밋 전 자가 리뷰 | 모든 커밋 직전 | 컨벤션, 프로세스, 보안 |
| PR 휴먼 리뷰 | PR 머지 전 | **6개 전체** |
| 적대적 크로스 리뷰 | 아래 필수 트리거 참조 | 6개 전체 (전문 프로파일) |

**통과 기준**: 연속으로 **두 번 클린 패스**가 나와야 리뷰 완료. 매 패스마다 progress file의 `review_evidence[]`에 읽은 파일 목록과 함께 기록.

## 차원 1: 아키텍처 일관성

코드가 올바른 레이어에 있고 통신 계약을 준수하는지 확인합니다.

### 레이어 정의

| 레이어 | 목적 | 포함하면 안 되는 것 |
|--------|------|-------------------|
| `shell` | UI, Tauri IPC 핸들러, 아바타, 패널 시스템 | 비즈니스 로직, DB 직접 접근, LLM 호출 |
| `agent` | LLM 연결, 툴 실행, 서브에이전트 오케스트레이션 | UI 상태, WebSocket 서버 |
| `gateway` | 항시 실행 데몬, 채널 라우팅, 메모리, 스킬 | UI 직접 인터랙션, shell 타입 import |
| `os` | BlueBuild 레시피, systemd, OS 설정 | 앱 레벨 코드, 런타임 로직 |

### 통신 계약

- **shell → agent**: stdio JSON lines만 (Tauri가 agent를 자식 프로세스로 spawn)
- **shell → gateway**: WebSocket `ws://localhost:18789` — 직접 import 금지
- **Tauri 이벤트 페이로드**: `event.payload`는 사용 전 반드시 타입 가드로 검증
- **신규 Tauri 커맨드**: `shell/src-tauri/capabilities/*.json`에 등록 필수
- **패널 툴**: `descriptor.tools[]`에 선언 후 `sendPanelSkills()`로 전송 — App.tsx에 하드코딩 금지

### 체크리스트

- [ ] 올바른 레이어에 배치됐는가?
- [ ] 크로스 레이어 직접 import가 없는가?
- [ ] stdio / WebSocket 프로토콜 변경이 하위 호환인가?
- [ ] 신규 Tauri 커맨드가 capabilities에 등록됐는가?
- [ ] 패널 툴이 descriptor에 선언됐는가?
- [ ] 6개월 후에도 이해 가능한가?

## 차원 2: 코딩 컨벤션

### 포매터 (Biome)

```
들여쓰기: tab | 따옴표: double | 세미콜론: always | trailing_comma: all | 줄 길이: 100
```

검사: `pnpm biome check --diagnostic-level=error` — **커밋 전 반드시 실행**.

### 네이밍

| 대상 | 규칙 | 예시 |
|------|------|------|
| 파일 / 디렉터리 | kebab-case | `avatar-canvas.tsx` |
| 컴포넌트 / 클래스 | PascalCase | `AvatarCanvas` |
| 함수 / 훅 | camelCase | `getBridgeForPanel` |
| 상수 | UPPER_SNAKE_CASE | `NAIA_WIDTH_DEFAULT` |
| 타입 / 인터페이스 | PascalCase, `I-` 접두사 없음 | `PanelDescriptor` |
| Rust 파일 | snake_case | `lib.rs` |

### 로깅

**금지**: `console.log`, `console.warn`, `console.error`, `eprintln!`, `println!`

**필수 (TypeScript)**: `./lib/logger`의 `Logger.info/warn/error/debug`

**빌드 타임 규칙**: 로깅은 버그 발생 후가 아니라 **구현 중에** 추가. 모든 새 async 연산, 상태 전환, 외부 호출에 Logger 항목 필수.

### 체크리스트

- [ ] Biome lint + format 통과?
- [ ] 새 파일이 kebab-case?
- [ ] `console.log` / `eprintln!` 없음?
- [ ] 새 async 연산 / 상태 전환 / 외부 호출에 `Logger.debug` 있음?
- [ ] Named import 사용 — 새 default export 없음?
- [ ] 무방비 `any` 또는 `!` 단언 없음?
- [ ] 시스템 경계에서만 에러 처리?

## 차원 4: 보안 위협

**중요**: 체크리스트 통과만으로는 부족 — 실제 공격 가능성을 평가해야 함. 심각도 ≥ medium 발견 시 적대적 크로스 리뷰 필수.

| 위협 | 벡터 | 심각도 | 완화 방법 |
|------|------|--------|----------|
| IPC 인젝션 | 외부 채널 → gateway → Tauri emit → 검증 없이 `event.payload` 사용 | **높음** | 모든 `event.payload` 타입 가드 |
| CSP 우회 | LLM 출력을 `innerHTML`로 렌더링 → XSS | **높음** | LLM 출력을 원시 HTML로 절대 렌더링 금지 |
| 크리덴셜 노출 | `Logger.info`에 `naiaKey` / `apiKey` 포함 | **높음** | 크리덴셜 필드를 Logger에 절대 전달 금지 |
| 권한 티어 우회 | 티어 미지정 신규 툴 → 무제한 실행 | **높음** | 모든 툴 정의에 `permission_tier` 명시 |
| 프롬프트 인젝션 | 외부 콘텐츠가 LLM 시스템 프롬프트 탈취 | **중간** | 외부 콘텐츠 명확히 구분; 프롬프트 변경 리뷰 |
| 경로 탐색 | `skill_file_read`에 `../../etc/passwd` | **높음** | `~/` 샌드박스 내 경로 검증 |
| 에셋 프로토콜 범위 | `assetProtocol.scope`가 과도하게 넓어 의도치 않은 파일 노출 | **중간** | 최소 필요 경로만 |

### 체크리스트

- [ ] 모든 Tauri `event.payload`가 타입 가드됐는가?
- [ ] 비정제 콘텐츠의 `innerHTML` / `dangerouslySetInnerHTML` 없음?
- [ ] Logger 호출에 크리덴셜 필드 없음?
- [ ] 모든 신규 툴에 `permission_tier` 명시?
- [ ] LLM 시스템 프롬프트 변경이 인젝션 표면 관점에서 리뷰됐는가?
- [ ] 모든 파일 연산이 샌드박스 내 경로를 검증하는가?
- [ ] `assetProtocol.scope`가 불필요하게 확장되지 않았는가?
- [ ] 외부 채널 입력이 Tier 0-1로 제한됐는가?

## 차원 5: 실질 테스트

### 테스트 품질 규칙

테스트 결과를 신뢰하기 전:
1. 테스트 작성 → 단언문 리뷰 (올바른 타겟? 엣지 케이스? 충분히 엄격?)
2. 발견된 문제 수정 → 재리뷰 → **테스트 코드에 대해 연속 두 번 클린 패스**
3. 그 다음에 실행
4. 통과 후: 테스트가 구현 세부사항이 아닌 의도된 동작을 검증하는지 확인

### 유형별 필수 커버리지

**UI 컴포넌트**: 크래시 없이 렌더, 사용자 인터랙션 → 예상 상태 변화, 엣지 케이스 (빈 상태 / 로딩 / 에러)

**Agent 툴**: 정상 경로, 권한 티어 거부, 에러/타임아웃 처리

**Gateway 스킬**: 라우트 디스패치 정확성, 채널 격리

**Shell 시작**: 스플래시 표시, 스플래시 종료 후 메인 UI, 흰 화면 없음

### E2E 필수 경로 (모두 통과해야 함)

- 앱 실행 → 스플래시 → 메인 UI (흰 화면 없음)
- 채팅 메시지 전송 → LLM 응답 수신 (스트리밍)
- 툴 실행 → 권한 프롬프트 → 결과 반환
- 패널 전환 → 스킬 올바르게 업데이트
- 설정 변경 → 저장 → 재시작 시 반영

### 체크리스트

- [ ] 모든 새 동작에 테스트 추가?
- [ ] 단언문이 엄격한가 (정확히 일치, contains 대신)?
- [ ] 엣지 케이스 커버됨 (빈값, null, 에러, 타임아웃)?
- [ ] E2E 필수 경로가 실제 실행 앱에서 테스트됐는가?
- [ ] 테스트 코드 자체가 리뷰됐는가 (두 번 클린 패스)?
- [ ] 문서화 없이 스킵된 테스트 없음?
- [ ] 커버리지 목표 달성 (agent 80%, shell 70%, gateway 80%)?

## 차원 6: 요구사항 적합성

### 레벨

1. **기능적** — Issue에 명시된 것을 구현했는가?
2. **행동적** — 사용자 관점에서 올바르게 동작하는가?
3. **경험적** — 느낌이 맞는가? (UX, 성능, 시각적 완성도)
4. **비회귀** — 기존에 작동하던 것을 망가뜨리지 않았는가?

### UX 기준 (shell UI 변경 시)

스플래시 전환, 레이아웃 변경, 애니메이션, 아바타 렌더링, BGM 플레이어에 적용:

- 상태 전환 중 플래시/깜빡임 없음
- 다음 상태가 나타나기 전 애니메이션 완료 (겹침 없음)
- 콘텐츠 대기 중 로딩 상태 처리 (스피너 또는 스켈레톤)
- 에러 상태가 시각적으로 처리됨 (`console.error`만으로 끝내지 않음)
- 창 리사이즈 시 레이아웃 깨지지 않음
- 시작 시퀀스 전체에서 어두운 배경 유지

### 완료 기준

- [ ] 원래 Issue 재확인 — 모든 인수 기준 충족?
- [ ] 수동 우회 없이 사용자 가시적 결과 달성 가능?
- [ ] 비회귀: 기존 기능 여전히 동작?
- [ ] UX: 플래시 없음, 애니메이션 완료, 로딩 + 에러 상태 처리?
- [ ] 메인 스레드 새 블로킹 연산 없음?
- [ ] 부작용이 있다면 문서화?

## 크로스 리뷰 (적대적) 호출

스킬: `/cross-review`

### 필수 트리거

- 심각도 ≥ medium 보안 발견 시
- 신규 Tauri IPC 커맨드 또는 `listen()` 핸들러 추가 시
- 아키텍처 경계 변경 (새 크로스 레이어 통신 경로)
- CSP `connect-src`에 새 URL 추가 시
- LLM 시스템 프롬프트 또는 페르소나 메시지 변경 시
- 권한 모델 변경 (새 티어 또는 새 툴)

### 권장 트리거

- 세 레이어 모두 관련된 기능 (shell + agent + gateway)
- 새 패널 등록 (특히 keepAlive)
- 주요 UI 플로우 변경 (스플래시, 온보딩, 레이아웃 재설계)

### 리뷰 프로파일

| 프로파일 | 집중 영역 |
|---------|----------|
| Architecture | 레이어 위반, 계약 준수, 하위 호환성 |
| Security | IPC 인젝션, 크리덴셜, CSP, 권한 티어, 경로 탐색 |
| Code | 컨벤션 준수, 로깅 커버리지, 테스트 품질 |
| Requirements | 사용자 스토리 충족, UX 정확성, 비회귀 |

### 결과

| 결과 | 의미 | 조치 |
|------|------|------|
| Pass | 모든 프로파일: medium/high 발견 없음 | 머지 허용 |
| Conditional | low 발견만 있음 | 머지 전 수정, 재리뷰 불필요 |
| Fail | medium/high 발견 있음 | 수정 후 크로스 리뷰 재실행 |
