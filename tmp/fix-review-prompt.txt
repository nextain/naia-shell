The fix:
- attemptConnect captures `const sock = ws;` right after `ws = new WebSocket(wsUrl)`.
- Guards added: `if (sock !== ws) return;` at the top of ws.onmessage, ws.onerror, ws.onopen, and the 15s timeout callback (timeout also now calls sock.close() not ws?.close()). onopen sends setup on `sock` not the global `ws`.
- ws.onclose uses a DIFFERENT guard: `if (ws !== null && sock !== ws) return;` — because disconnect() sets the global `ws = null` during teardown, and the current socket's onclose must still settle the pending connect() promise (otherwise disconnect-during-connecting hangs). So onclose proceeds when ws===null (teardown) or sock===ws (current), and only skips when a newer non-null socket has superseded this one.

Context: disconnect() sets aborted=true, connectAbort.abort(), then `if (ws) { ws.close(); ws = null; }`. The retry loop catches attemptConnect rejections; `if (aborted) throw`.
