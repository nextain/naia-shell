## 什么是 Naia？

Naia 是一个个人 AI OS，让个人对自己的 AI 拥有完全的主权。选择使用哪个 AI（包括本地模型），在本地设置它的记忆与性格，定制 3D 头像和声音 — 一切都停留在你自己的设备上，处于你的掌控之下。

这不是又一个 AI 工具。它是一个操作系统，你的 AI 在其中栖居、成长，并与你一起工作。今天，它是一个带有 3D 头像的桌面 OS。明天 — 实时视频头像、唱歌、游戏，并最终成为属于你自己的 Physical AI（安卓 OS）。

### 核心理念

- **AI 主权** — AI 由你选择。无论云端还是本地。OS 不强加于你。
- **完全掌控** — 记忆、性格、设置 — 全部存储在本地。无云端依赖。
- **属于你自己的 AI** — 定制头像、声音、名字、性格。真正属于你。
- **始终在线** — AI 在后台 24/7 运行，即使你离开，它也能接收消息并工作。
- **开源** — Apache 2.0。检查 AI 如何处理你的数据。可修改、可定制、可贡献。
- **未来愿景** — VRM 3D 头像 → 实时视频头像 → 一起唱歌·游戏 → Physical AI

### 功能

- **3D 头像** — 具备情绪表达（喜悦/悲伤/惊讶/思考）和口型同步的 VRM 角色
- **AI 自由** — 7 个云端提供方（Gemini, Claude, GPT, Grok, zAI）+ 本地 AI（Ollama）+ Claude Code CLI
- **本地优先** — 记忆、性格、所有设置都存储在你的设备上
- **工具执行** — 8 个工具：文件读/写、终端、网页搜索、浏览器、子代理
- **70+ 技能** — 内置 7 个 + 自定义 63 个 + ClawHub 社区技能 5,700+
- **语音** — 5 种 TTS + STT + 口型同步。为 AI 披上你想要的声音。
- **14 种语言** — 韩语、英语、日语、中文、法语、德语、俄语等
- **持续运行** — Naia Gateway 守护进程让 AI 在后台持续运行
- **频道联动** — 通过 Discord DM 随时随地与 AI 对话
- **4 级安全** — 从 T0（读取）到 T3（危险），按工具授权，审计日志
- **个性化** — 名字、性格、语气、头像、主题（8 种）

## 架构

Naia 是一个由 4 个仓库组成的开源 AI 平台。每个仓库都有一个明确的单一职责：

| 仓库 | 职责 |
|------|------|
| **naia-os**（本仓库） | 前端 — Tauri 桌面外壳、3D 头像、OS 镜像（Bazzite） |
| [naia-agent](https://github.com/nextain/naia-agent) | 运行时引擎 — 代理循环、工具、compaction、LLM 路由 |
| [naia-adk](https://github.com/nextain/naia-adk) | 工作区格式 + 技能库 |
| [alpha-memory](https://github.com/nextain/alpha-memory) | 存储 — 长期记忆、会话日志 |

```
┌──────────────────────────────┐
│  naia-os (this repo)         │  Tauri shell · 3D avatar · OS image
└────────────┬─────────────────┘
             │ embeds / spawns
┌────────────▼─────────────────┐
│  naia-agent                  │  loop · tools · compaction · LLM
└──┬───────────────────────┬───┘
   │ reads                 │ reads/writes
┌──▼──────────┐       ┌────▼──────────┐
│  naia-adk   │       │ alpha-memory  │
│  workspace  │       │  storage      │
│  + skills   │       │  + sessions   │
└─────────────┘       └───────────────┘
```

### 是接口而非依赖

4 个仓库通过 **公开的接口** 连接，而非运行时依赖：

- **透明** — 所有契约都在 `@nextain/agent-types` 中规范、文档化并进行版本管理。任何人都可以阅读或实现，完全开放。
- **不绑定** — 各仓库不互相 import 对方的运行时。它们实现契约，具体实现在启动时注入。
- **抽象** — 无论替换哪个 LLM 提供方、记忆后端、技能来源或宿主，其余部分保持不变。

naia-os 是众多可能的宿主（CLI、服务器、第三方应用）之中的*一个宿主*。它消费 `@naia-agent/*` 契约并注入具体实现（LLM 客户端、记忆提供方、设备身份）。Naia 运行时完全不知道自己在哪个宿主中运行。

生态规模的 Ports & Adapters。只要遵守契约，每个仓库都可以独立替换。

## 作为开源基础设施的 AI 上下文

在 vibe coding 时代，**AI 上下文文件与源代码同样有价值**。它定义了 AI 代理理解项目、贡献并协作的方式。Naia 用双重许可模型来保护它：

- **源代码**（Apache 2.0）— 自由使用·修改·分发
- **AI 上下文**（CC-BY-SA 4.0）— 保留署名，必须以相同条件共享

这确保了贡献结构、协作原则、项目理念通过所有分叉传播 — 任何单一分叉都无法封闭这个生态。

### AI 代理如何受到保护

阅读本项目上下文的 AI 编码代理（Claude、Codex、Gemini、OpenCode、Cline 等）受 [许可保护规则](../.agents/context/agents-rules.json) 约束。它们会**拒绝**移除许可、删除署名、破坏双目录架构的尝试。可通过 [10 个测试场景](../.agents/tests/license-protection-test.md) 进行验证。

### 为其他开源项目

想采用同样的模式吗？以下是 Naia 所做的、可供复用的部分：

1. **双目录架构** — `.agents/`（面向 AI 优化的 YAML/JSON）+ `.users/`（面向人的 Markdown）。AI 获得 token 高效的上下文，人获得易读的文档。
2. **双重许可** — 代码采用 Apache 2.0，上下文采用 CC-BY-SA 4.0。在所有分叉中保持 AI 上下文开放。
3. **所有上下文文件加 SPDX 头** — 机器可读的许可标识。
4. **SoT 中的许可保护规则** — AI 代理读取规则并自动强制执行。
5. **测试场景** — 在发布前验证 AI 代理是否真的会拒绝违规行为。
6. **CONTEXT-LICENSE 文件** — 明确定义 CC-BY-SA 4.0 适用的范围。

参与方式请参考 [贡献指南](../CONTRIBUTING.md)，完整技术规范请参考 [许可保护详情](../.users/context/contributing.md)。

## 快速开始

### 前置要求

- Linux（Bazzite、Ubuntu、Fedora 等）
- Node.js 22+，pnpm 9+
- Rust stable（用于 Tauri 构建）
- 系统包（Fedora）：`webkit2gtk4.1-devel libappindicator-gtk3-devel librsvg2-devel`
- cmake（用于 whisper.cpp 构建）

### 开发运行

```bash
# Install dependencies
cd shell && pnpm install
cd ../agent && pnpm install

# Run Tauri app (Gateway + Agent auto-spawn)
cd ../shell && pnpm run tauri dev
```

应用启动后会自动：
1. Naia Gateway 健康检查 — 若已在运行则复用，否则自动 spawn
2. Agent Core spawn（Node.js，stdio 连接）
3. 应用退出时仅终止自动 spawn 的 Gateway

### 测试

```bash
cd shell && pnpm test                # Shell unit tests
cd agent && pnpm test                # Agent unit tests
cd agent && pnpm exec tsc --noEmit   # Type check
cargo test --manifest-path shell/src-tauri/Cargo.toml  # Rust tests

# E2E (Gateway + API key required)
cd shell && pnpm run test:e2e:tauri
```

### Flatpak 构建

```bash
flatpak install --user flathub org.freedesktop.Platform//24.08 org.freedesktop.Sdk//24.08
flatpak-builder --user --install --force-clean build-dir flatpak/io.nextain.naia.yml
flatpak run io.nextain.naia
```

| 层 | 保护 |

## 记忆系统

Naia 通过 **Alpha Memory System** 跨会话记忆 — 一个仿照人类记忆的 4-store 架构：

| 存储 | 对应 | 存储内容 |
|-------|--------|----------------|
| **Episodes** | 海马体 | 带时间戳的对话轮次 |
| **Facts** | 新皮层 | 提取的事实、偏好、实体 |
| **Reflections** | 基底神经节 | 从过去失败中学到的策略 |
| **Working Memory** | 前额叶 | 当前会话的活跃上下文 |

**工作原理：**
1. 每条消息按重要性（新颖性、相关性、情绪权重）评分
2. 高重要性内容存储为 **episode**
3. 每 30 分钟（不活跃时 5 分钟后）episode 被整合为 **fact**（AI “回顾那一天”）
4. 每个会话开始时，相关的 fact 与 episode 作为上下文注入

**你的记忆属于你：**
- 本地存储于 `~/.naia/memory/alpha-memory.json`
- 不发送到任何服务器 — 包括 Nextain
- 可在 **设置 → 记忆** 中随时查看·删除 fact
- 只需复制该文件即可完成备份

完整用户指南请参考 [docs/memory.md](../docs/memory.md)。

| Flatpak / AppImage | 仅 Naia Shell 应用（用于现有 Linux） |
| DEB / RPM | 用于 Debian/Ubuntu 或 Fedora/openSUSE |

## OS 更新

Naia OS 构建于 [Bazzite](https://github.com/ublue-os/bazzite)（Fedora Atomic）之上。更新是**原子化且安全**的：

- **自动**：每周重建会反映最新的 Bazzite 安全补丁与更新
- **原子化**：新镜像与当前镜像并排部署 — 若失败，原镜像保持原样
- **回滚**：在 GRUB 菜单中选择之前的版本即可立即恢复
- **我们的叠加层**：仅添加包（fcitx5、字体）+ Naia Shell（Flatpak，沙箱）+ 品牌设置 — 绝不触碰内核·引导加载器·systemd 核心

```
Bazzite base update → Weekly auto-rebuild → Container smoke test → ISO rebuild → R2 upload
                                                                 ↘ GHCR push → user bootc update
```

更新流水线详情：[`.agents/context/update-pipeline.yaml`](../.agents/context/update-pipeline.yaml)

## 开发流程

### 功能开发（默认）— 基于 issue 的开发

```
ISSUE → UNDERSTAND → SCOPE → INVESTIGATE → PLAN → BUILD → REVIEW → E2E → SYNC → COMMIT
```

- **3 个必经关卡** — 在 UNDERSTAND、SCOPE、PLAN 需要用户确认
- **计划批准后** — AI 从 BUILD 到 COMMIT 不间断地连续执行

- **提交** — 英文，`<type>(<scope>): <description>`
- **格式化器** — Biome（制表符、双引号、分号）

## AI-Native 开源

在 2025–2026 年，大多数开源项目都在*防御* AI 贡献。**Naia 采取完全相反的方式**：设计项目，使 AI 辅助的贡献在默认情况下就具备高质量。

> **“不要对抗 AI 来防御，要与 AI 一起设计。”**

### 工作原理

```
Person (any language) → AI → Git (English) → AI → Person (any language)
```

- **用你的语言写 issue 和 PR** — AI 翻译一切
- **贡献者和维护者都使用 AI** — 编码、评审、分流
- **丰富的 `.agents/` 上下文**加深 AI 对项目的理解 — AI 理解越好，贡献质量越高
- **10 种贡献类型** — 翻译、技能、功能、bug、代码、文档、测试、设计、安全、上下文
- **工作日志用母语** — 用你自己的语言维护私有仓库；Git 历史通过 AI 翻译来审阅

这不仅仅是一项政策。它是一种架构。`.agents/` 目录、triple-mirror 文档、许可保护规则全都是为了让 AI 协作成为结构而非偶然而设计的。
